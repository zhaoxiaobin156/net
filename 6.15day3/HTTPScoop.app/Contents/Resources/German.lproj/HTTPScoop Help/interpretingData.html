Inhaltsverzeichnis

Die Daten die HTTPScoop anzeigt verstehen

HTTPScoop bietet mehrere Ansichten für aufgezeichnete HTTP-Kommunikation in den verschiedenen Registern des Fensters HTTP-Anfrage-Details. Dieses Fenster öffnet sich, wenn sie auf eine Anfrage doppelklicken.

  • Zusammenfassung
  • Kopfzeilen
  • POST (Hex)
  • Antwort (Text)
  • Antwort (Hex)
  • Bild
  • TCP/IP

Zusammenfassung

Die Zusammenfassung zeigt allgemeine Informationen der Anfrage und den aktuellen Status wie die angefragte URL, die Antwortzeile, den Content-Type und die Größe der Anfrage/Antwort.

Die Anfrage-Parameter werden extrahiert (bei GET-Anfragen aus der URL; bei POST-Anfragen mit dem Content-Type application/x-www-form-urlencoded aus den POST-Daten) um eine gute Übersicht zu gewährleisten.

Kopfzeilen

Jede Anfrage und jede Antwort kann Kopfzeilen enthalten. Anfrage-Kopfzeilen von Webbrowsern enthalten typischerweise Informationen über Sprachpräferenzen, Browsertyp, akzeptierte Dateitypen usw. Antwort-Kopfzeilen von einem Server enthalten Informationen über die Größe und Art der zurückgegebenen Inhalte sowie Informationen darüber wie (und ob) die Inhalte zwischengespeichert werden dürfen.

Ein weiterer wichtiger Anwendungsfall von Kopfzeilen sind Cookies. Cookies erlauben der Website anzufragen, ihr innerhalb des Browsers Daten zuzuordnen und diese bei zukünftigen Anfragen zu übermitteln. Dies erlaubt z.B. das Speichern von Benutzereinstellungen, die Analyse von Surfgewohnheiten etc. Cookies können außerdem verwendet werden um eine Sitzung zu verfolgen, so dass der Server verschiedene Anfragen die vom selben Browser ausgehen einander zuordnen kann. Halten Sie Ausschau nach Cookie-Anfragekopfzeilen sowie Set-cookie-Antwortkopfzeilen.

Weitere Informationen über Cookies erhalten Sie in der RFC 2965 - Mechanismen zur Statusverwaltung bei HTTP (englisch), allgemeine Informationen über Anfrage-Kopfzeilen erhalten Sie in der HTTP-Spezifikation (englisch).

POST (Hex)

Der Anfragekörper einer POST-Anfrage kann hier als Hexdump betrachtet werden. Bitte beachten Sie, dass es bei der Anzeige von großen Anfragen zu Verzögerungen kommen kann.

Antwort (Text)

Zeigt den Antwortkörper als Klartext an (entpackt - falls gzip- oder deflate-komprimiert - sowie zusammengesetzt - falls die Antwort aus mehreren Teilen bestand).

Das zerstückelte Übertragen (Chunking) von Daten wird oft beim Übertragen von Inhalten verwendet, deren Länge vorher nicht bekannt ist. Anstatt eine fixe Länge für den Inhalt anzugeben, sendet der Server mehrere Stücke mit Daten vor denen jeweils eine Längenbeschreibung steht. Das Ende der Antwort wird durch ein Teilstück mit der Länge 0 signalisiert. Da HTTPScoop die Teile automatisch aneinanderreiht bevor der Antworttext angezeigt wird, ist nicht ersichtlich ob Chunking eingesetzt wurde. Dies kann aber in der TCP/IP-Ansicht beobachtet werden.

Für die Content-Types text/html und text/xml wird eine einfache Syntayhervorhebung verwendet, welche Elemente, Attribute, Attributwerte und Kommentare hervorhebt.

Antwort (Hex)

Der Körper einer Antwort kann hier als Hexdump betrachtet werden. Bitte beachten Sie, dass es bei der Anzeige von großen Antworten zu Verzögerungen kommen kann. Wie bei der Ansicht der Antwort als Text, werden die Daten auch hier (wo anwendbar) nach der Dekomprimierung und nach dem Zusammensetzen angezeigt.

Bild

Das Register Bild zeigt zurückgegebene Bilder an (skaliert, falls sie zu groß sind). Bitte beachten Sie, dass die Content-Type-Kopfzeile korrekt gesetzt sein muss, damit ein Bild angezeigt wird.

TCP/IP

Diese Ansicht ermöglicht Ihnen die komplette Aktivität der TCP-Verbindung einzusehen, über die die Anfrage gemacht wurde. Dies erlaubt Zugriff auf die Rohdaten die über die Leitung gesendet wurden, vor Dekomprimierung, Zusammensetzen mehrerer Chunks, etc. Dies kann nützlich sein um Protokollfehler von HTTP-Clients/-Servern zu finden.

Sequenz- und Quittierungsnummern werden zusammen mit dem Status der TCP-Flags SYN, ACK, RST und FIN angezeit. Die Paketkörper werden als Hexdump dargestellt.

Die HTTP-Spezifikation erlaubt das Senden von mehreren Anfragen über eine TCP-Verbindung ohne auf eine Antwort zu warten. Aus diesem Grund könnte der erste TCP-Anfrage- bzw. Antwortkörper von einer anderen als der gewählten HTTP-Anfrage sein.
