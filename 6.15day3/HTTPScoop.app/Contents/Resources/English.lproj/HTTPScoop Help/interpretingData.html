Understanding the information HTTPScoop presents to you

HTTPScoop provides a number of views of captured HTTP conversations which are shown in different tabs in the HTTP Request Detail pop-up window. This window appears when you double-click a request.

  • Summary
  • Headers
  • POST Hex
  • Response Text
  • Response Hex
  • Response Image
  • TCP/IP

Summary

The request summary tab gives general information about the request and its current state: the URL, response line, content type and request/response size.

For ease of viewing, request parameters are broken out of the URL for requests made using the GET method, or out of the POST data for POST requests using the content-type application/x-www-form-urlencoded.

Headers

Each request and response may have associated header information. Request headers from web browsers typically include information about language preferences, browser type, acceptable file types and so on. Typical response headers from a server include information about the size and type of content being returned and how the content should be cached on the client, if at all.

Another important use of headers is to support cookies. Cookies allow a site to request that information is associated with it within the web browser and returned when future requests are made. This allows user preferences to be stored, browsing habits to be monitored etc. Cookies may also be used to track browser sessions so that the server is able to see that different requests originate from the same browser window. Look out for Cookie request headers and Set-cookie response headers.

For more information on cookies, see RFC 2965 - HTTP State Management Mechanism or for request headers in general, see the HTTP specification.

POST Hex

The body of a POST request can be viewed in hex dump form on this tab. Note that there may be a delay in rendering dumps for large POSTs.

Response Text

Shows the response body in text form after decompression (if gzip or deflate content encoding is used) and “de-chunking”.

Chunking is a common technique for handling dynamic content, the size of which is not known beforehand. Rather than specifying a fixed content length the server sends “chunks” of data, with each chunk preceded by a length descriptor. The end of the response is indicated by a chunk size of zero. Because de-chunking is performed by HTTP Scoop automatically before the response text is shown, you won’t see chunking information in the response text. It can, however, be observed in the TCP/IP view.

Basic syntax colouring is performed for text/HTML and text/XML content types with elements, attributes, attribute values and comments highlighted.

Response Hex

The body of a response can be viewed in hex dump form on this tab. Note that there may be a delay in rendering dumps for large responses. Like Response Text, this is a view of the response data after decompression and de-chunking where applicable.

Response Image

The Response Image tab shows returned images, scaled down to fit the view where appropriate. Note that the response Content-Type header must be set correctly for the image to be shown.

TCP/IP

This view allows you to see all activity on the TCP connection over which a request has been made. This gives you access to the raw data being sent over the wire, prior to any decompression, de-chunking etc. It can be useful in diagnosing protocol errors in custom HTTP clients or servers.

TCP sequence numbers and acknowledgments are shown along with the state of the TCP flags SYN, ACK, RST and FIN. Packet bodies are shown in a hex dump format.

The HTTP specification allows that multiple HTTP requests may be sent over the same TCP connection without waiting for a response. For this reason, the first request/response TCP packet body you see may not be for the HTTP request you selected.
