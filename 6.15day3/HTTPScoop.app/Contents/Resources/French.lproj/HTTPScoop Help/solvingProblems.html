Résolution des problèmes

Si vous avez un problème qui ne peut pas être résolu grâce à ces suggestions, veuillez contacter support@tuffcode.com et nous essayerons de vous aider.

Catégories de problèmes :

  • Requêtes manquantes
  • "Réinitialisée par un pair"
  • "Fermée par un pair"
  • "Trafic trop rapide : quelques paquets perdus"

Requêtes manquantes

Si vous ne voyez pas les requêtes HTTP espérées, vérifiez d'abord que vous bien sélectionné l'adaptateur réseau approprié dans le menu déroulant en haut à droite. Vérifiez aussi que vous n'avez pas été trop restrictif dans vos choix de ports dans les préférences.

Spécifier un nom d'hôte dans la liste d'hôtes des préférences peut causer quelques problèmes. Premièrement, les autres ressources, comme les images, peuvent être stockées sur un autre serveur. Ensuite, certaines serveurs utilisent plusieurs IP pour le même nom d'hôte, afin de réduire la charge du serveur. L'adresse IP enregistrée par HTTPScoop risque alors de ne pas être la même que celle à laquelle le client accède.

"Réinitialisée par un pair"

Ce message de statut indique que le client ou le serveur ont forcé la fermeture de la connexion TCP utilisée pour la requête.

Les connexions peuvent être réinitialisées pour différentes raisons. Cela peut être parce qu'une erreur a été détectée dans les données reçues.

"Fermée par un pair"

Ce message de statut indique que la connexion TCP utilisée pour la conversation HTTP a été fermée avant que la réponse soit complètement lue. Cela peut être du à une erreur du serveur ou réseau.

Ce message peut aussi apparaître si certains paquets ont été perdus. Regardez alors "Trafic trop rapide : quelques paquets perdus".

"Trafic trop rapide : quelques paquets perdus"

Ce message apparaissant dans la barre de statut en haut indique que le système a ignoré certains paquets parce que la charge réseau ou CPU était trop grande.

Un client HTTP comme un navigateur ne souffre pas de ce problème parce qu'il utilise TCP pour contrôler le flot de données. Mais un observateur passif comme HTTPScoop doit être capable de soutenir le flux réseau afin de reconstruire précisément la conversation.

Le problème arrive le plus souvent dans ces situations :

  • Haute charge CPU : si l'ordinateur sur lequel HTTPScoop est lancé est occupé par beaucoup de taches, le système accorde moins de temps à la capture et le décodage des paquets et peut commencer à la ignorer.
  • Haute charge réseau : s'il y a lien rapide entre le client et le serveur, HTTPScoop peut ne pas tenir le coup.
  • En mode promiscueux : s'il y a un fort trafic passant par l'adaptateur réseau qui est destiné à d'autres machines et que vous êtes en mode promiscueux, le nombre de paquets a analysé peut devenir trop grand.

La meilleure façon d'éviter ce problème est de s'assurer que vous surveillez uniquement les ports TCP appropriés et que vous utilisez le mode promiscueux uniquement quand nécessaire. Vous devriez aussi réduire l'activité des autres applications.

Notez qu'après que cet avertissement ait été affiché, vous ne pourrez plus enregistrer correctement le corps de la réponse. Si le fichier a été capturé avant l'erreur et que son statut est "Terminé", il ne devrait pas y avoir de problèmes. Sinon, videz la liste, recommencez la requête et réessayez d'enregistrer le fichier.
